### DD-1: Decorator governance mechanism (registry lookup, not attrs)

**V2 Phase 0.1.3 Option (c) is REJECTED** — three experts (PE, SecArch, SAD) independently verified that `WardlineAnnotation.attrs` is never populated with `_wardline_transition` for bare `@validates_shape`, because `_extract_decorator_attrs` at `src/wardline/scanner/discovery.py:387-408` only handles `ast.Call` decorators (i.e., `@validates_shape(arg=val)`), not `ast.Name` decorators (i.e., bare `@validates_shape`).

**The correct mechanism (DD-1)** is to consult the decorator object via the `wardline.decorators` namespace:

```python
def _get_validation_decorator_transition(canonical_name: str) -> tuple[TaintState, TaintState] | None:
    """Look up a decorator's declared trust transition from the decorator object itself.

    Returns (source, target) when the canonical name is a registered validator that
    carries a `_wardline_transition` attribute; returns None otherwise. The decorator
    object is the authoritative source because `_wardline_transition` is set at
    decoration time by `wardline.decorators.authority`, not encoded in the AST.
    """
    import wardline.decorators as wd
    dec = getattr(wd, canonical_name, None)
    if dec is None:
        return None
    transition = getattr(dec, "_wardline_transition", None)
    if not isinstance(transition, tuple) or len(transition) != 2:
        return None
    return transition
```

**Why this works**: `canonical_name` is populated from the registry during annotation discovery (`discovery.py:361`). The registry is a trusted source-controlled artefact. Trusting a registry-canonical name to resolve to a registry-owned decorator object is honest single-layer defense with the registry-sync test (Chunk 1 §1.6) as the governance-layer cross-check.

**Attack surface**: An adversary would need to either (a) edit `wardline.core.registry` directly (source-controlled), or (b) shadow the `wardline.decorators` module. Both are out-of-band attacks that the decorator-check cannot reasonably prevent — they require commit access and the same CODEOWNERS review as the rule itself.

### DD-3: Shared helper on `rules/base.py` with parameterised decorator set

```python
def _has_validation_boundary_decorator(
    self,
    overlay_scope: str | None,
    *,
    allowed: frozenset[str] = _VALIDATION_DECORATORS,
) -> bool:
    """Check if the enclosing function carries a validation-boundary decorator.

    The `allowed` parameter lets individual rules pass a different decorator set
    (e.g., PY-WL-007 plugin contracts). The `overlay_scope` parameter supports
    two sentinel cases:
      - None or "" → unscoped, no path check (used when the caller cannot
        determine a relevant scope, e.g., PY-WL-003 fallback)
      - Any other string → must resolve via `path_within_scope(file, scope)`

    Never uses `"."` as a scope sentinel — `"."` resolves via `Path.resolve()`
    to the process CWD and produces false negatives in test harnesses where
    file paths are absolute test fixtures outside the CWD.
    """
    from wardline.manifest.scope import path_within_scope
    if self._context is None or not self._current_qualname:
        return False
    annotations = self._context.annotations_map.get(self._current_qualname, ())
    if not annotations:
        return False
    for ann in annotations:
        if ann.canonical_name not in allowed:
            continue
        transition = _get_validation_decorator_transition(ann.canonical_name)
        if transition is None or transition[0] is not TaintState.EXTERNAL_RAW:
            continue
        if overlay_scope:
            if not path_within_scope(self._file_path, overlay_scope):
                continue
        return True
    return False
```

**Scope sentinel rule**: `None` or `""` means "skip scope check" (caller has no scope). `"."` is **forbidden** as a scope value — reject it with a `ValueError` at call time to prevent SAD B2 recurring.

**Import rule**: `path_within_scope` must be a module-top import in `base.py`, not inline. If the cycle SAD predicts exists, it must be broken by moving `path_within_scope` to `wardline.core` or inverting the dependency. Fail loudly on `ImportError`, never silent False.

```markdown
### 18. Silent rule narrowing through implementation drift

Wardline rule implementations can silently drift from their spec definitions
through incremental code changes that contradict a MUST clause without updating
the spec, without writing an ADR, and without breaking existing tests. The
Wardline architecture does not prevent this at a structural level: rule code
and spec text are independently maintained, and tests are written by the same
author who writes the code (no adversarial assertion-vs-spec enforcement).

**Mechanism**: An author (human or automated) edits a rule to fix a failing
test or add a perceived improvement. The edit contradicts a MUST clause in
the spec. New unit tests are added to validate the new behavior. Existing
tests pass because they were not written to assert the MUST clause directly.
No ADR is written because the author believes the change is incremental.
The spec is not updated. The drift persists until an external review detects
the gap.

**Compensating controls in this codebase**:

1. MUST-clause traceability test (`tests/conformance/test_must_clause_traceability.py`)
   — every MUST clause in spec normative chapters must be cited by at least
   one test via a `SPEC_REF` class attribute, and every `SPEC_REF` must resolve
   to an existing MUST-id.
2. `Spec-Impact` commit trailer on any rule-module PR, with differential
   corpus gate that auto-promotes the label when any cell verdict flips.
3. Bidirectional CODEOWNERS + branch protection requiring different-actor
   approval (subject to DD-7 residual gap for this project's single-reviewer
   state).
4. Adversarial panel review for any rule semantics change.

**Residual risk after controls**: A drift that does not flip any corpus cell
(e.g., widening the set of permitted validators without affecting existing
specimens) can still escape the automated gates. The adversarial review
mechanism is the last line of defense and relies on human discipline.

This risk is illustrated by commit 7caf751 (2026-04-09), which loosened
PY-WL-001 `schema_default()` governance from function-level to file-level
in contradiction of §A.3 clause 3. The drift was detected 3 days later
through a test-failure investigation (not through the compensating controls
above, which did not exist at the time). Restoration is documented in ADR-004.
```

```python
"""Bidirectional traceability between spec MUST clauses and test assertions."""
import re
from pathlib import Path

SPEC_PATH = Path("docs/spec/wardline-02-A-python-binding.md")
TEST_DIR = Path("tests")

MUST_ID_PATTERN = re.compile(r"<!-- must-id: (\w+\.\w+\.\w+) -->")
SPEC_REF_PATTERN = re.compile(r"SPEC_REF\s*=\s*['\"]([^'\"]+)['\"]")

def test_must_clause_parser_self_test(tmp_path):
    """Known-good fixture test — proves the regex catches real anchors."""
    fixture = tmp_path / "fixture.md"
    fixture.write_text("Text <!-- must-id: TEST.0.0 --> more text")
    ids = set(MUST_ID_PATTERN.findall(fixture.read_text()))
    assert ids == {"TEST.0.0"}, f"Parser failed self-test: {ids}"

def test_every_must_clause_has_a_citing_test():
    """Every MUST clause in spec must be cited by at least one test."""
    spec_text = SPEC_PATH.read_text()
    must_ids = set(MUST_ID_PATTERN.findall(spec_text))
    assert must_ids, "Spec has no must-id anchors — parser or spec is broken"

    cited = set()
    for test_file in TEST_DIR.rglob("test_*.py"):
        for match in SPEC_REF_PATTERN.findall(test_file.read_text()):
            for id_match in re.findall(r"[A-Z]\.\d+\.\d+", match):
                cited.add(id_match)

    uncited = must_ids - cited
    assert not uncited, f"MUST clauses with zero citing tests: {uncited}"

def test_every_spec_ref_resolves_to_a_real_must_id():
    """Every SPEC_REF in test code must cite a real must-id."""
    spec_text = SPEC_PATH.read_text()
    must_ids = set(MUST_ID_PATTERN.findall(spec_text))

    dangling = set()
    for test_file in TEST_DIR.rglob("test_*.py"):
        for match in SPEC_REF_PATTERN.findall(test_file.read_text()):
            for id_match in re.findall(r"[A-Z]\.\d+\.\d+", match):
                if id_match not in must_ids:
                    dangling.add((test_file.name, id_match))

    assert not dangling, f"SPEC_REFs citing non-existent MUST-ids: {dangling}"
```

```python
"""Compare per_cell verdicts between baseline and current corpus verify output.

A cell flip (PASS → FAIL or FAIL → PASS) requires a Spec-Impact trailer
in the PR body. A new cell (absent in baseline) is reported but does not
fail the check (new specimens are expected on rule-module PRs).
"""
import argparse
import json
import sys

def main():
    parser = argparse.ArgumentParser()
    parser.add_argument("--baseline", required=True)
    parser.add_argument("--current", required=True)
    parser.add_argument("--fail-on-flip", action="store_true")
    args = parser.parse_args()

    with open(args.baseline) as f:
        baseline = json.load(f)
    with open(args.current) as f:
        current = json.load(f)

    def cell_key(cell):
        return (cell.get("rule"), cell.get("taint_state"), cell.get("exceptionability"))

    baseline_map = {cell_key(c): c for c in baseline.get("per_cell", [])}
    current_map = {cell_key(c): c for c in current.get("per_cell", [])}

    flips = []
    new_cells = []
    for key, current_cell in current_map.items():
        baseline_cell = baseline_map.get(key)
        if baseline_cell is None:
            new_cells.append(key)
            continue
        if baseline_cell.get("cell_verdict") != current_cell.get("cell_verdict"):
            flips.append((key, baseline_cell.get("cell_verdict"), current_cell.get("cell_verdict")))

    if new_cells:
        print(f"New cells (not compared): {new_cells}")
    if flips:
        print(f"Cell verdict flips: {flips}")
        if args.fail_on_flip:
            sys.exit(1)
    else:
        print("No cell verdict flips.")

if __name__ == "__main__":
    main()
```

**Scope:** Three-oracle retrospective scan per DD-8, per-cell corpus verify, fingerprint regeneration with supersedes trail, self-hosting scan, final acceptance.

**Depends on:** ALL previous chunks (needs fixed rule, fixed tests, CI gates, docs).

**Critical adversarial findings addressed:**

- Co-masking detection (SecArch FLAW-C)
- Fingerprint regeneration supersedes trail (IRAP R2-02, R2-03)

### 6.1 Oracle 1: Pickaxe search

- [ ] `git log -p -S 'schema_default(' --since=2026-04-09 --until=2026-04-12 -- src/ tests/ corpus/` — record output.
- [ ] For each hit, inspect manually. Any new `schema_default` call in the window is a candidate exploit artifact.
- [ ] Record findings in ADR-004 §Consequences (even if empty — "no hits" is defensible evidence).

### 6.2 Oracle 2: Co-masking detection

- [ ] `git log -p --since=2026-04-09 --until=2026-04-12 -- src/` — search for any commit that adds BOTH:
  - a `@validates_shape` or `@validates_external` decorator
  - a `schema_default(` call site
  - in the same diff
- [ ] Flag every match for manual review. This catches the attack where an adversary adds a fake decorator and a fabrication in the same commit.
- [ ] Record findings in ADR-004 §Consequences.

### 6.3 Oracle 3: Differential SARIF scan

- [ ] For each commit in the window:
  1. `git worktree add /tmp/retro-<sha> <sha>`
  2. `( cd /tmp/retro-<sha> && uv run wardline scan src/ --output /tmp/broken-<sha>.sarif )` — runs the broken rule from that commit
  3. Copy the fixed rule files (`py_wl_001.py`, `base.py`, etc.) from HEAD into `/tmp/retro-<sha>/src/wardline/scanner/rules/`
  4. `( cd /tmp/retro-<sha> && uv run wardline scan src/ --output /tmp/fixed-<sha>.sarif )` — runs the fixed rule
  5. Diff SARIF outputs by `(file, function_qualname, rule_id, taint_state)` — qualname-keyed, not line-keyed
  6. `git worktree remove /tmp/retro-<sha> --force`
- [ ] Any finding in the diff where `fixed \ broken` is non-empty is a candidate exploit artifact.
- [ ] Record in ADR-004 §Consequences.
